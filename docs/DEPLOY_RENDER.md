# Deploying NexaFlow to Render + Vercel

This is the simplest production path. Render's "Blueprint" feature reads
the `render.yaml` at the repo root and provisions everything in one click.

| Layer | Hosted on | Why |
|---|---|---|
| Next.js web frontend | **Vercel** | Already deployed at `whatsapp-api-ruddy-alpha.vercel.app` |
| Express API + Socket.io | **Render Web Service** | Long-lived process; native Docker support |
| BullMQ workers | **Render Background Worker** | 8 workers run continuously, scale separately |
| Postgres | **Render Postgres** | Managed, in-region |
| Redis | **Render Key Value** | Managed, BullMQ-compatible |

This split solves the "Can't reach the server" error: Vercel hosts the web,
Render hosts everything else, and `NEXT_PUBLIC_API_URL` connects them.

## One-time setup (~15 minutes)

### 1. Push the blueprint to GitHub

Already done — `render.yaml` is on the `codex/nexaflow-v2-platform` branch.

### 2. Create the Blueprint on Render

1. Go to https://dashboard.render.com → **New** → **Blueprint**
2. Connect your GitHub account → select `Dropdead6420/whatsapp-api`
3. Branch: `codex/nexaflow-v2-platform`
4. Render reads `render.yaml` and shows a preview of what it'll create:
   - Web service: `nexaflow-api`
   - Worker: `nexaflow-worker`
   - Database: `nexaflow-db` (Postgres)
   - Key Value: `nexaflow-redis`
5. Click **Apply**.

Render starts provisioning. The first build takes 4-7 minutes because it
has to install dependencies and run Prisma generate. Subsequent deploys
take ~90 seconds.

### 3. Fill in the secrets

While the build is running, click into the `nexaflow-api` service →
**Environment** tab. You'll see the keys marked `sync: false` waiting
for values:

| Key | Where to get it | Required? |
|---|---|---|
| `META_APP_ID` | https://developers.facebook.com → your app → App ID | Yes (for WhatsApp) |
| `META_APP_SECRET` | Same page, App Secret | Yes |
| `WHATSAPP_WEBHOOK_TOKEN` | Pick any random string; Meta will verify | Yes |
| `META_PHONE_NUMBER_ID` | Your WABA phone number id from Meta Business Manager | Yes |
| `META_BUSINESS_ACCOUNT_ID` | Your WABA id | Yes |
| `OPENAI_API_KEY` | https://platform.openai.com/api-keys | Optional (KB embeddings) |
| `ANTHROPIC_API_KEY` | https://console.anthropic.com | Optional (AI features) |
| `SENTRY_DSN` | https://sentry.io project settings | Optional |

`JWT_SECRET` and `TENANT_TOKEN_ENCRYPTION_KEY` are **auto-generated by Render**
(see `generateValue: true` in `render.yaml`) — you don't set them.

Save. Render redeploys the service automatically.

### 4. Run database migrations

Once the API service shows "Live", open it → **Shell** tab → run:

```bash
node node_modules/prisma/build/index.js migrate deploy \
  --schema packages/db/prisma/schema.prisma
```

You should see output ending with:

```
Applying migration `0_init`
Applying migration `1779274127_add_branding_demo`
Applying migration `20260522090000_add_knowledge_base`
Applying migration `20260523120000_add_ai_agent`

All migrations have been successfully applied.
```

### 5. Verify the API is up

The blueprint sets `healthCheckPath: /api/v1/ready`. Render's dashboard
should show a green checkmark. From your terminal:

```bash
curl https://nexaflow-api.onrender.com/api/v1/ready
```

Expected:

```json
{
  "status": "ready",
  "mode": "api",
  "services": [
    { "name": "postgres", "ok": true },
    { "name": "redis", "ok": true }
  ]
}
```

If `postgres` or `redis` is `false`, check the service's **Logs** tab.

### 6. Point Vercel at the new API

In the Vercel dashboard:

1. Open the `whatsapp-api-ruddy-alpha` project
2. **Settings → Environment Variables**
3. Add `NEXT_PUBLIC_API_URL` = `https://nexaflow-api.onrender.com`
   for **Production**, **Preview**, AND **Development** environments
4. **Deployments → ⋯ → Redeploy** (env-var changes alone don't rebuild;
   Next has to bake the new value into the JS bundle)

After the Vercel redeploy finishes (~60s), hard-reload `https://whatsapp-api-ruddy-alpha.vercel.app/login` (Cmd+Shift+R). The "Can't reach the server" error will be gone.

### 7. Update CORS if your Vercel domain is custom

If your final web URL is *not* `whatsapp-api-ruddy-alpha.vercel.app`:

- Render dashboard → `nexaflow-api` → Environment → change `WEB_URL`
- Save (Render will restart the service)

## Important gotcha: Render Free tier sleeps

The `render.yaml` ships with `plan: free` on the web service. The free
tier **sleeps after 15 minutes of inactivity** and takes ~30s to wake on
the next request. This means:

- **First login of the day will hang for 30s.**
- **Background workers won't sleep** (they're on `plan: starter`), so
  campaigns and webhooks keep firing — only the *human-facing API* sleeps.

For production, change the API service's `plan` to `starter` ($7/mo) so
it stays warm 24/7:

```yaml
services:
  - type: web
    name: nexaflow-api
    plan: starter   # was: free
```

Push the change; Render auto-redeploys.

## Scaling

```bash
# Render has no CLI for this — use the dashboard.
# Per service: Settings → Scaling → set instance count or upgrade plan.
```

Render scales:

- **Web service** → multiple instances behind a load balancer (sticky
  sessions on for Socket.io — Render does this automatically when you
  add instances).
- **Worker** → multiple instances pull from the same Redis queue;
  BullMQ handles work distribution.
- **Postgres** → upgrade plan (Standard adds read replica).
- **Redis** → upgrade plan; for Cluster mode, you'd self-host instead.

## Troubleshooting

### Login page still shows "Can't reach the server"

1. Hard-reload (Cmd+Shift+R) — the old JS bundle is cached
2. Open browser devtools → Network tab → submit the login form. The failing request reveals the URL the bundle is calling. If it's still `localhost`, the Vercel redeploy didn't run.
3. `curl https://nexaflow-api.onrender.com/api/v1/ready` from your terminal. 503 = DB/Redis not connected; check Render logs.
4. `curl -I -H "Origin: https://whatsapp-api-ruddy-alpha.vercel.app" https://nexaflow-api.onrender.com/api/v1/auth/login` — if `Access-Control-Allow-Origin` is missing, `WEB_URL` on Render is wrong.

### "Application failed to respond" 502

The free-tier web service is asleep. Hit `/api/v1/ready` once to wake it,
then retry. Upgrade to `starter` plan to prevent this entirely.

### Worker isn't processing campaigns

`nexaflow-worker` → **Logs**. Look for:

- `[shutdown] received SIGTERM` → Render restarted it; usually harmless
- `BullMQ: Error: connect ECONNREFUSED` → Redis link broke; rerun the blueprint to reattach
- Anything with `prisma`: schema drift; run `migrate deploy` from the Shell tab

### Migrations needed after a schema change

```bash
# In the nexaflow-api service Shell tab:
node node_modules/prisma/build/index.js migrate deploy \
  --schema packages/db/prisma/schema.prisma
```

Run this **after** Render finishes deploying the new code so the new
migration files are present.

## Why this works (vs. trying to run the API on Vercel)

Vercel functions die after 60s (Hobby) or 5min (Pro). The NexaFlow API has:

- **8 BullMQ workers** that need to run continuously
- **Socket.io realtime layer** that holds persistent WebSocket connections
- **Express middleware chain** with raw-body capture for Meta webhook signatures

Render's Web Service + Background Worker model is exactly the long-lived
process abstraction this codebase was designed for. No refactor needed.
